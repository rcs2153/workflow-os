# 0.1.0-preview.1 Release Checklist

Release under review: **Workflow OS `0.1.0-preview.1` local kernel preview**

This checklist prepares a human maintainer to tag and publish release notes for the public local-kernel preview. Do not use this checklist to claim production readiness.

## Pre-Tag Checks

- [ ] Confirm the release is described as a public local-kernel preview.
- [ ] Confirm docs do not claim production readiness.
- [ ] Confirm docs do not claim unsupported GitHub/Jira write behavior, CI, SaaS, or generic HTTP adapters exist.
- [ ] Confirm docs do not claim distributed workers exist.
- [ ] Confirm docs do not claim a production database backend exists.
- [ ] Confirm docs do not claim hosted SaaS, UI, marketplace, or package registry support exists.
- [ ] Confirm docs do not claim Level 3/4 autonomy is enabled.
- [ ] Confirm CLI JSON output is marked experimental.
- [ ] Confirm `serde_yaml` / `unsafe-libyaml` risk is documented as accepted for trusted local project files only.
- [ ] Confirm `YAML-001` remains tracked in `ROADMAP.md`.
- [ ] Confirm `workflow-os doctor state` is documented as inspection-only, not repair.
- [ ] Confirm no Git tag has been created accidentally before maintainer approval.

## Vertical-Slice Smoke Test

Build the CLI:

```sh
cargo build -p workflow-cli --bin workflow-os
```

Use a temporary local state root:

```sh
STATE_DIR="$(mktemp -d /tmp/workflow-os-preview.XXXXXX)"
```

Run:

```sh
target/debug/workflow-os --project-dir examples/vertical-slice-approval --state-dir "$STATE_DIR" validate
target/debug/workflow-os --project-dir examples/vertical-slice-approval --state-dir "$STATE_DIR" --mock-all-local-skills run ex/review
target/debug/workflow-os --project-dir examples/vertical-slice-approval --state-dir "$STATE_DIR" --mock-all-local-skills approve <run-id> <approval-id> --actor user/release-reviewer --reason release-smoke-test
target/debug/workflow-os --project-dir examples/vertical-slice-approval --state-dir "$STATE_DIR" status <run-id>
target/debug/workflow-os --project-dir examples/vertical-slice-approval --state-dir "$STATE_DIR" inspect <run-id>
target/debug/workflow-os --project-dir examples/vertical-slice-approval --state-dir "$STATE_DIR" doctor state
```

Expected results:

- [ ] `validate` prints `Project is valid.`
- [ ] `run` enters `WaitingForApproval` and prints `run_id` and `approval_id`.
- [ ] `approve` completes the run.
- [ ] `status` reports `Completed` and `terminal: true`.
- [ ] `inspect` shows event history from `RunCreated` through `RunCompleted`.
- [ ] `doctor state` reports healthy state.

## Package Publication Decision

For `0.1.0-preview.1`, package publication is **not automatic**.

Current repository state:

- CI does not run `cargo publish`.
- CI does not run `npm publish`.
- The root npm package is marked `"private": true`.
- The TypeScript SDK package is not marked private, but there is no automatic publish workflow.
- Rust crates have package metadata and versions but no automatic publishing workflow.

Default preview decision:

- [ ] Do not publish crates to crates.io unless maintainers explicitly approve a separate publication plan.
- [ ] Do not publish npm packages unless maintainers explicitly approve a separate publication plan.
- [ ] Treat the GitHub release source archive as the public preview artifact unless maintainers decide otherwise.

If maintainers later decide to publish packages:

- [ ] Confirm package names and ownership.
- [ ] Confirm README/package metadata.
- [ ] Confirm included files.
- [ ] Confirm dry-run output.
- [ ] Confirm credentials are configured securely.
- [ ] Record the publication decision in release notes.

## Post-Release Verification

After the tag and GitHub release:

- [ ] Confirm the GitHub release is marked pre-release.
- [ ] Confirm release text says local-kernel preview.
- [ ] Confirm release text says not production-ready.
- [ ] Confirm release text says no unsupported adapters, no write-capable adapters, and no distributed runtime.
- [ ] Confirm source archives are available.
- [ ] Clone from the release tag into a fresh directory.
- [ ] Run `cargo test --workspace`.
- [ ] Run `npm ci`.
- [ ] Run `npm run check`.
- [ ] Run the vertical-slice quickstart from the release tag.

## Rollback Or Unpublish Guidance

If the GitHub release text overclaims:

- [ ] Edit the GitHub release immediately.
- [ ] Keep the tag unless code or artifacts are wrong.
- [ ] Add a corrective note in the release body.

If the tag points at the wrong commit:

- [ ] Do not force-move the tag silently.
- [ ] Delete the GitHub release draft if needed.
- [ ] Coordinate with maintainers before deleting and recreating the tag.
- [ ] Prefer a follow-up `0.1.0-preview.2` if the tag may already have been fetched.

If a package was accidentally published:

- [ ] Follow the registry-specific unpublish/deprecate policy.
- [ ] Publish a corrective advisory in the GitHub release.
- [ ] Add a changelog note explaining the mistake.
- [ ] Rotate any accidentally exposed credentials immediately.
