# 0.2.0-preview.1 Release Checklist

Release under review: **Workflow OS `0.2.0-preview.1` public read-only integration preview**

This checklist is for a human maintainer preparing the release. Do not tag or publish from automation unless maintainers explicitly approve.

## 1. Pre-Tag Checks

- [ ] Confirm the release posture is read-only only.
- [ ] Confirm live provider use remains opt-in.
- [ ] Confirm normal CI remains fixture-first and credential-free.
- [ ] Confirm no production readiness, write support, hosted service, distributed runtime, Level 3/4 autonomy, OAuth, webhook, production backend, or domain-pack claim appears in public-facing docs.
- [ ] Confirm accepted live-smoke limitations are visible in release notes and GitHub release draft.
- [ ] Confirm Jira sandbox token rotation has been completed or explicitly tracked before announcement.
- [ ] Confirm GitHub `main` branch/ruleset protection is configured and verified before public promotion.

## 2. Required Commands

Run from the repository root:

```sh
cargo fmt --all --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
RUSTDOCFLAGS="-D warnings" cargo doc --workspace --no-deps
cargo audit
npm ci
npm run check
npm run check:contracts
npm run check:integrations
npm audit --audit-level=moderate
```

Expected result: all commands pass. Live provider tests must remain ignored by default in `cargo test --workspace`.

## 9. Tag Creation Step

Only a human maintainer should create and push the tag:

```sh
git status --short
git tag -a v0.2.0-preview.1 -m "Workflow OS 0.2.0-preview.1 public read-only integration preview"
git push origin v0.2.0-preview.1
```

Do not create the tag if the working tree is dirty, required checks failed, or limitations are not reflected in release materials.

## 10. Package Publication Decision

For `0.2.0-preview.1`, package publication is **not automatic**.

- [ ] Decide whether crates should be published.
- [ ] Decide whether the TypeScript package should be published.
- [ ] If not publishing, state that the GitHub release/tag is the preview artifact.
- [ ] If publishing, run dry-run/package checks first and document publication ownership.

## 11. Post-Release Verification

- [ ] GitHub release page links to release notes and known limitations.
- [ ] Tag points at the intended commit.
- [ ] CI is green on the release commit.
- [ ] README renders correctly.
- [ ] Example commands still match the release docs.
- [ ] No provider credentials or sensitive provider payloads were added to release artifacts.

## 12. Rollback / Unpublish Guidance

If the tag is wrong:

- Do not rewrite public history casually.
- Prefer publishing a corrective preview release unless maintainers agree the tag must be deleted.
- If deletion is required, record why in release notes or maintainer notes.

If a package is accidentally published:

- Follow the package registry's unpublish/deprecate policy.
- Rotate any exposed credentials immediately.
- Publish a security note if user-impacting artifacts were exposed.

If release notes overclaim:

- Correct the release notes immediately.
- Add a changelog correction.
- Do not continue promotion until the public posture is accurate.
