# Workflow OS 0.1.0-preview.1 Release Notes

## Release Title

Workflow OS `0.1.0-preview.1` local kernel preview

## Release Posture

Workflow OS `0.1.0-preview.1` is a **public local-kernel preview**.

It is a serious local-first foundation for defining, validating, testing, executing, governing, observing, and eventually scaling AI-driven workflows. It is intended for builders, evaluators, and contributors who want to inspect the kernel architecture, run the local vertical slice, exercise the validator and CLI, and help harden the next phases.

This is not a production release. It is not an adapter-complete release. It is not a distributed-runtime release. It is not a hosted product release. It is not a UI release. It is not a marketplace release. It is not a Level 3/4 autonomy release.

## What Is Included

- Rust-owned canonical core primitives.
- Versioned YAML project and spec loading for `workflow-os.yml`, workflows, skills, policies, and tests.
- Checked-in JSON Schemas under `schemas/v0/`.
- Deterministic project loading and semantic validation with structured diagnostics.
- Event-sourced workflow run model with schema version, workflow version, spec content hash, sequence numbers, transition validation, and deterministic rehydration.
- Local filesystem development backend with duplicate detection, append validation, idempotency records, local locking, corruption detection, and state inspection.
- `workflow-os doctor state` for read-only local state inspection.
- Single-step local executor for explicitly registered local skills.
- Approval pause/resume and CLI approval grant/denial.
- Bounded retry, escalation, cancellation, and timeout representation for the local runtime path.
- Conservative policy and capability model that denies unknown actions, unknown capabilities, Level 3/4 execution, adapter invocation, and external writes by default.
- Runtime audit and observability foundations with local sinks and durable local policy audit records.
- CLI commands: `validate`, `run`, `status`, `approve`, `inspect`, and `doctor`.
- TypeScript SDK helpers for generating Workflow OS specs; the SDK does not execute workflows.
- Schema/SDK/example contract checks against Rust CLI validation.
- Vertical-slice approval example using an explicitly enabled deterministic local mock skill.
- Generic adapter contracts for future integrations without real external API clients.

## Quickstart

Build the CLI:

```sh
cargo build -p workflow-cli --bin workflow-os
```

Validate the vertical-slice example:

```sh
target/debug/workflow-os --project-dir examples/vertical-slice-approval validate
```

Run the local approval workflow with explicit deterministic mock skill handlers:

```sh
target/debug/workflow-os --project-dir examples/vertical-slice-approval --mock-all-local-skills run ex/review
```

The run pauses and prints a `run_id` and `approval_id`.

Approve and resume:

```sh
target/debug/workflow-os --project-dir examples/vertical-slice-approval --mock-all-local-skills approve <run-id> <approval-id> --actor user/example-approver --reason reviewed-local-example
```

Inspect the completed run:

```sh
target/debug/workflow-os --project-dir examples/vertical-slice-approval inspect <run-id>
```

The example uses an explicitly enabled deterministic local mock skill. The CLI does not execute arbitrary declared `local/*` skills by default.


Run the vertical-slice smoke path with a temporary state root:

```sh
STATE_DIR="$(mktemp -d /tmp/workflow-os-preview.XXXXXX)"
target/debug/workflow-os --project-dir examples/vertical-slice-approval --state-dir "$STATE_DIR" validate
target/debug/workflow-os --project-dir examples/vertical-slice-approval --state-dir "$STATE_DIR" --mock-all-local-skills run ex/review
target/debug/workflow-os --project-dir examples/vertical-slice-approval --state-dir "$STATE_DIR" --mock-all-local-skills approve <run-id> <approval-id> --actor user/release-reviewer --reason release-smoke-test

target/debug/workflow-os --project-dir examples/vertical-slice-approval --state-dir "$STATE_DIR" inspect <run-id>
```

## Security Notes

The security posture is coherent for a local-first kernel preview, not production deployment.

Confirmed controls include:

- Rust crates deny unsafe code.
- Secret-like spec keys and values are rejected by loader/parser paths.
- Conservative policy denies unknown capabilities, adapter invocation, external writes, and Level 3/4 execution by default.
- Approval gates execute before approval-sensitive local skill invocation.
- Runtime audit uses references and summaries rather than raw sensitive payloads by default.
- Dependency audits pass for this preview.

This release does not include enterprise RBAC, IdP integration, a real secret provider, production audit export, tamper-evident event logs, sandboxed untrusted skill handlers, or production backend security controls.

## YAML Parser Accepted Risk

Maintainer decision for `0.1.0-preview.1`: Workflow OS accepts `serde_yaml` for trusted local project files only.

This is a documented preview risk because `serde_yaml` is deprecated and depends on `unsafe-libyaml`. Specs in this release are treated as local project files authored and reviewed in Git, not untrusted remote input, uploaded SaaS content, webhook payloads, or adversarial documents.

Workflow OS must not claim hardened malicious-spec parsing, parser sandboxing, or production-grade parser isolation for `0.1.0-preview.1`. `YAML-001` tracks replacing or isolating YAML parsing before any production-readiness or adversarial-input hardening claim.

## Experimental CLI JSON Note

`--json` output exists for preview automation, smoke tests, and contract checks. In `0.1.0-preview.1`, CLI JSON is experimental and is not a versioned stable machine-output contract. Fields, enum formatting, and response shapes may change in later v0 releases with release notes.

Before CLI JSON is promoted to a stable contract, Workflow OS must define a versioned response envelope, document compatibility rules, and add contract tests for every command that supports JSON output.

## Upgrade And Migration Notes

This is the first public local-kernel preview, so there is no prior public release migration.

Local event JSON created by pre-preview development snapshots without `schema_version` is incompatible with the current immutable run identity contract. Workflow OS intentionally does not infer or default missing schema versions during rehydration. Keep old local state roots for investigation and use a fresh state root for preview runs unless a future migration tool is introduced.
