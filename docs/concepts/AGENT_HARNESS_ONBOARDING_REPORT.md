# Agent Harness Onboarding Report

Report date: 2026-06-16

## 1. Executive Summary

Workflow OS now documents the kernel-governed agent onboarding path as a P0 roadmap priority.

The implemented documentation makes the intended operating model explicit: users should connect a coding agent such as Codex or Claude Code to the local Workflow OS kernel and instruct the agent to use the kernel as the governing layer. Manual YAML authoring remains supported, but it is no longer presented as the whole user experience.

This phase adds a canonical quickstart, root agent instructions, README guidance, dogfood guidance, user-guide links, roadmap placement, and an implementation plan. It does not implement runtime harness auto-generation, CLI scaffold commands, automatic local check execution, writes, side-effect modeling, reasoning lineage, recursive agents, agent swarms, hosted runtime behavior, or release posture changes.

## 4. Onboarding Summary

The onboarding path now tells users to:

1. build the local CLI;
2. validate the dogfood project;
3. start a governed dogfood run;
4. paste the canonical agent prompt into Codex or Claude Code;
5. approve kernel checkpoints when required;
6. let the agent execute repository work inside the governed phase boundary;
7. inspect the run and review the final report.

The key product framing is:

```text
Agent executes. Workflow OS governs.
```

## 5. Safety Boundary Summary

The instructions explicitly prohibit:

- bypassing validation, policies, approvals, checks, or scope limits;
- inventing workflow state, approvals, evidence, audit events, reports, validation results, or command outputs;
- treating model self-review as deterministic governance;
- claiming recursive agents, agent swarms, hosted runtime behavior, writes, production self-hosting, or Level 3/4 autonomy.

## 6. Future Automation Summary

The docs identify a future setup helper such as:

```sh
workflow-os init-agent-harness
```

That future phase would generate `AGENTS.md`, a session prompt, and starter governed workflow instructions. It remains unimplemented and must not silently enable command execution, writes, hosted behavior, schema changes, or higher autonomy.
